서버 사이드 렌더링(Server-Side rendering, SSR)
- 전통적인 MVC 웹 애플리케이션
- Node.js EJS 뷰 엔진을 쓰는 방식
- 동작 방식
	1. 브라우저가 요청을 보냄
	2. 서버가 DB에서 정보를 조회
	3. EJS 템플릿 + 데이터를 합쳐서 완성된 HTML을 만듦
	4. 완성된 HTML을 브라우저로 전송
	5. 브라우저는 HTML 그려서 화면에 보여줌
- 장점
	- 구조 단순
	- 빠르게 페이지 만들기 좋음
클라이언트 사이드 렌더링(CSR, SPA)
- 프론트엔드와 백엔드를 분리한 아키텍처
- API 서버 + 프론트(순수 JS, React/Vue) 분리
- 동작 방식
	1. js 코드 안에서 fetch 또는 axios 등으로 Node.js 서버의 REST API를 호출
	2. 프론트에서 JSON 데이터를 받아서 DOM을 직접 만들거나 React/Vue 컴포넌트로 렌더링
- 장점
	- 프론트와 백엔드가 역할이 명확하게 분리
	- API 재사용성이 높음(모바일 앱, 웹 서비스, 외부 공개 API 등으로 확장하기 좋음)
	- 프론트 프레임워크(React / Vue 등)와 연동이 좋음

웹소켓(WebSocket)
웹소켓은 http 위에서 연결을 시작한 뒤, 그 연결을 업그레이드하여 클라이언트와 서버가 하나의 tcp 연결로 실시간 양방향 통신을 할 수 있게 해주는 기술

양방향 통신
클라이언트에서 서버로, 서버에서 클라이언트로 언제든지 서로에서 메세지를 보낼 수 있음

지속적인 연결
http 처럼 요청-응답 후 끊어지지 않고 연결을 계속 유지하면서 수시로 데이터를 주고받을 수 있음

통신 방식
1. 초기에는 http 요청으로 "웹소켓으로 업그레이드 해줘"라고 요청(handshake)
2. 이후에는 http 가 아니라 WebSocket 프로토콜로 변경
3. 연결이 유지된 상태에서 서버가 먼저 메시지를 푸시할 수 있음

Node.js + WebSocket 의 장점
1. 동시 접속자 많은 시스템에 유리
   - Node.js 자체가 많은 연결을 이벤트 위주로 처리하는데 최적화 되어 있음
   - 채팅 서버, 게임 서버, 실시간 모니터링 서버 등에 많이 사용
2. JavaScript 풀스텍
   - 클라이언트(브라우저)도 자바스크립트, 서버도 자바스크립트 개발 모두 수월
   - WebSocket 메시지에 JSON 을 사용하면 서버와 클라이언트 간 데이터 교환이 직관적

구현 방식
1. 순수 WebSocket 프로토콜 라이브러리 사용
   ws 패키지
2. Socket.IO 사용(WebSocket + 폴백 + 추가 기능 포함)
   방/네임스페이스, 재연결, 브로드케스트 등 포함
